# 边端向量库选型方案

## 一、核心需求

- ✅ 嵌入式（无需单独服务）
- ✅ 跨平台（Windows/macOS/Linux/Android）
- ✅ 轻量（二进制文件 < 50MB）
- ✅ 易集成（最好纯 Go）

---

## 二、方案对比

### 方案1：SQLite + sqlite-vec（原方案）

**技术栈**：
- SQLite（C 库）
- sqlite-vec（C 扩展）
- 通过 CGO 调用

**优势**：
- ✅ 数据和向量在同一个 SQLite 文件
- ✅ 可以用 SQL 查询向量
- ✅ 成熟的 SQLite 生态

**劣势**：
- ❌ 需要 CGO（跨平台编译复杂）
- ❌ sqlite-vec 相对较新（2023年发布）
- ❌ Android 编译可能有问题

**适用场景**：个人电脑（Windows/macOS/Linux）

---

### 方案2：modernc.org/sqlite + 纯 Go 向量实现（推荐）

**技术栈**：
- modernc.org/sqlite（纯 Go 的 SQLite 实现）
- 自己实现向量检索（暴力搜索或简单 IVF）

**优势**：
- ✅ 纯 Go，无需 CGO
- ✅ 跨平台编译简单（`go build` 即可）
- ✅ 二进制文件小
- ✅ Android/iOS 完全支持

**劣势**：
- ⚠️ 需要自己实现向量检索算法
- ⚠️ 性能可能不如 C 实现的 sqlite-vec

**实现思路**：
```
SQLite 存储：
- memories 表：存储元数据（text, summary, entities 等）
- vectors 表：存储向量（memory_id, vector_blob）

向量检索：
- 从 vectors 表读取所有向量到内存
- 纯 Go 实现余弦相似度计算
- 暴力搜索（< 1万条记忆，< 50ms）
- 或实现简单的 IVF 索引（分桶）
```

**性能评估**：
- 1000 条记忆：< 10ms
- 5000 条记忆：< 30ms
- 10000 条记忆：< 50ms
- 对个人使用完全够用

---

### 方案3：BoltDB + 纯 Go 向量库

**技术栈**：
- BoltDB（纯 Go KV 数据库）
- 自己实现向量存储和检索

**优势**：
- ✅ 纯 Go
- ✅ 简单高效

**劣势**：
- ❌ 没有 SQL 查询能力
- ❌ 全文检索需要自己实现
- ❌ 开发工作量大

**不推荐**：失去了 SQLite 的 FTS5 全文检索能力

---

### 方案4：混合方案（灵活）

**技术栈**：
- SQLite（存储元数据和文本）
- 向量存储在单独的二进制文件（mmap）
- 纯 Go 向量检索

**优势**：
- ✅ 纯 Go
- ✅ 灵活，可以单独优化向量存储
- ✅ 保留 SQLite 的 SQL 和 FTS5 能力

**劣势**：
- ⚠️ 需要管理两个文件
- ⚠️ 实现稍复杂

**文件结构**：
```
~/.memory-rag/
├── memory.db          # SQLite（元数据 + 文本）
└── vectors.bin        # 向量二进制文件（mmap）
```

## 三、推荐方案

### 最佳选择：方案2（modernc.org/sqlite + 纯 Go 向量）

**理由**：
1. **纯 Go**：跨平台编译零问题
2. **简单**：只需一个 SQLite 文件
3. **够用**：个人数据量小，暴力搜索足够快
4. **可扩展**：未来可以优化为 IVF 或 HNSW

### 实现策略

**阶段1：暴力搜索（MVP）**
- 直接实现余弦相似度计算
- 遍历所有向量，找 Top-K
- 代码量 < 100 行
- 性能：1万条 < 50ms

**阶段2：优化（可选）**
- 实现 IVF（倒排文件索引）
- 将向量分成 N 个桶（如 100 个）
- 只搜索最相关的几个桶
- 性能提升 10-100 倍

**阶段3：高级优化（可选）**
- 实现 HNSW（分层可导航小世界图）
- 或集成 Faiss（如果愿意用 CGO）

## 四、纯 Go 向量检索实现思路

### 4.1 数据结构

```go
// 向量存储在 SQLite BLOB 字段
CREATE TABLE vectors (
    memory_id TEXT PRIMARY KEY,
    vector BLOB,  -- []float32 序列化
    dimension INTEGER
);
```

### 4.2 检索流程

```
1. 查询向量化：query → []float32
2. 从 SQLite 读取所有向量
3. 计算余弦相似度
4. 排序取 Top-K
5. 返回 memory_id 列表
6. 根据 memory_id 查询完整记忆
```

### 4.3 性能优化技巧

**技巧1：向量归一化**
- 存储时归一化向量（L2 norm = 1）
- 余弦相似度 = 点积（无需除法）

**技巧2：并行计算**
- 使用 goroutine 并行计算相似度
- 适合多核 CPU

**技巧3：缓存热点向量**
- 将常用向量缓存在内存
- 减少 SQLite 读取

**技巧4：SIMD 加速**
- 使用 Go 的 SIMD 库（如 gonum）
- 向量运算加速 2-4 倍

## 五、方案对比总结

| 维度 | sqlite-vec (CGO) | modernc.org/sqlite (纯Go) | 混合方案 |
|------|-----------------|--------------------------|---------|
| 跨平台编译 | ⚠️ 复杂 | ✅ 简单 | ✅ 简单 |
| Android 支持 | ⚠️ 困难 | ✅ 完美 | ✅ 完美 |
| 性能 | ✅ 最快 | ⚠️ 稍慢 | ⚠️ 稍慢 |
| 开发难度 | ✅ 简单 | ⚠️ 需实现向量检索 | ⚠️ 需实现向量检索 |
| 二进制大小 | ⚠️ 较大 | ✅ 小 | ✅ 小 |
| 维护成本 | ⚠️ 依赖外部库 | ✅ 自主可控 | ✅ 自主可控 |

**推荐**：modernc.org/sqlite + 纯 Go 向量实现

- 或考虑回退到 sqlite-vec（仅限个人电脑）
