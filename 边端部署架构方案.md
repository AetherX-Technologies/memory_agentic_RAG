# HybridMem-RAG 边端部署架构方案

## 一、核心定位

**个人知识中台 - 边端优先**

- ✅ 个人电脑：Windows/macOS/Linux 一键运行
- ✅ 手机端：Android/iOS 轻量部署
- ✅ 无需 Docker：单一二进制文件 + 嵌入式数据库
- ✅ 开箱即用：下载即用，零配置启动

---

## 二、技术架构调整

### 2.1 放弃 PostgreSQL，改用 SQLite

**原方案问题**：
- PostgreSQL 需要单独安装和运行
- 手机端几乎不可能装 PostgreSQL
- 对普通用户不友好

**新方案**：SQLite + sqlite-vec

| 特性 | PostgreSQL + pgvector | SQLite + sqlite-vec |
|------|----------------------|---------------------|
| 安装 | 需要单独安装服务 | 嵌入式，无需安装 |
| 部署 | 需要配置、启动服务 | 单文件数据库 |
| 手机端 | ❌ 几乎不可能 | ✅ 完全支持 |
| 向量检索 | ✅ pgvector | ✅ sqlite-vec |
| 全文检索 | ✅ tsvector | ✅ FTS5 |
| 性能 | 高（服务端） | 中等（嵌入式） |
| 适用场景 | 服务器、企业 | 个人设备、边端 |

**结论**：边端场景下，SQLite 是更好的选择。

### 2.2 单一二进制部署

**架构**：
```
memory-rag (单一可执行文件)
├── 内嵌 HTTP Server
├── 内嵌 MCP Server
├── SQLite 数据库引擎
├── sqlite-vec 向量扩展
└── 配置文件（可选）
```

**数据文件**：
```
~/.memory-rag/
├── memory.db          # SQLite 数据库文件
├── config.yaml        # 配置文件（可选）
└── logs/              # 日志目录
```

**启动方式**：
```bash
# 下载对应平台的二进制
# Windows: memory-rag-windows.exe
# macOS: memory-rag-macos
# Linux: memory-rag-linux
# Android: memory-rag-android

# 直接运行
./memory-rag serve

# 输出：
# ✓ SQLite 数据库已初始化: ~/.memory-rag/memory.db
# ✓ HTTP API 启动: http://localhost:8080
# ✓ MCP Server 就绪
# ✓ 浏览器插件可连接
```

## 三、跨平台部署方案

### 3.1 个人电脑（Windows/macOS/Linux）

**部署步骤**：
1. 下载对应平台的二进制文件
2. 双击运行（或命令行运行）
3. 浏览器访问 `http://localhost:8080` 验证

**优势**：
- 无需安装任何依赖
- 无需配置数据库
- 开箱即用

### 3.2 Android 手机

**方案A：Termux（推荐）**
1. 安装 Termux（开源终端模拟器）
2. 下载 `memory-rag-android` 二进制
3. 运行：`./memory-rag serve`
4. 手机浏览器访问 `http://localhost:8080`

**方案B：Android 原生应用**
- 用 Go Mobile 编译成 Android AAR 库
- 包装成原生 Android 应用
- 提供图形界面

### 3.3 iOS 手机

**方案A：iSH（iOS 上的 Linux 环境）**
1. 安装 iSH（App Store 可下载）
2. 下载 Linux 版本二进制
3. 运行服务

**方案B：云端部署 + 移动端访问**
- 在个人电脑上运行服务
- 手机通过局域网访问
- 或者部署到个人云服务器

**方案C：iOS 原生应用（长期）**
- 用 Go Mobile 编译
- 包装成 iOS 应用
- 需要 Apple Developer 账号

## 四、数据库方案对比

### 4.1 SQLite + sqlite-vec 能力评估

| 功能 | PostgreSQL + pgvector | SQLite + sqlite-vec | 是否满足需求 |
|------|----------------------|---------------------|-------------|
| 向量存储 | ✅ | ✅ | ✅ |
| 向量检索（ANN） | ✅ HNSW | ✅ 暴力搜索/IVF | ✅ 个人数据量小，暴力搜索够用 |
| 全文检索 | ✅ tsvector | ✅ FTS5 | ✅ |
| JSONB 支持 | ✅ | ✅ JSON1 扩展 | ✅ |
| 事务支持 | ✅ | ✅ | ✅ |
| 并发写入 | ✅ 高 | ⚠️ 低（单写入） | ✅ 个人使用，并发低 |
| 数据量上限 | TB 级 | 理论 281TB | ✅ 个人数据 GB 级足够 |

**结论**：SQLite + sqlite-vec 完全满足边端场景需求。

### 4.2 数据模型调整

**SQLite 表结构**：
```sql
-- 记忆表
CREATE TABLE memories (
    id TEXT PRIMARY KEY,
    raw_text TEXT NOT NULL,
    summary TEXT,
    vector BLOB,  -- sqlite-vec 向量类型
    entities TEXT,  -- JSON 数组
    topics TEXT,    -- JSON 数组
    category TEXT,
    scope TEXT,
    importance REAL,
    uri TEXT,
    parent_uri TEXT,
    level INTEGER,
    consolidated INTEGER DEFAULT 0,
    created_at INTEGER,
    updated_at INTEGER
);

-- 向量索引（sqlite-vec）
CREATE VIRTUAL TABLE vec_memories USING vec0(
    memory_id TEXT PRIMARY KEY,
    embedding FLOAT[1024]
);

-- 全文检索索引（FTS5）
CREATE VIRTUAL TABLE fts_memories USING fts5(
    memory_id,
    content,
    content=memories,
    content_rowid=rowid
);

-- 连接关系表
CREATE TABLE memory_connections (
    from_id TEXT,
    to_id TEXT,
    relationship TEXT,
    strength REAL,
    created_at INTEGER,
    PRIMARY KEY (from_id, to_id)
);
```

## 五、核心算法适配

### 5.1 混合检索（Vector + BM25）在 SQLite 上的实现

**向量检索**：
- 使用 sqlite-vec 的 `vec_search()` 函数
- 小数据量（< 10万条）：暴力搜索足够快（< 100ms）
- 大数据量：sqlite-vec 支持 IVF 索引

**BM25 全文检索**：
- 使用 SQLite FTS5（Full-Text Search）
- 内置 BM25 算法
- 性能优秀，支持中文分词

**RRF 融合**：
- 在应用层实现（Go 代码）
- 分别查询向量和 BM25 结果
- 按 RRF 公式合并排序

### 5.2 12阶段评分管道

**实现方式**：
- Stage 1-4（检索与融合）：数据库层 + 应用层
- Stage 5-12（评分调整）：纯应用层实现
- Stage 10（关联图谱加权）：需要查询 connections 表

**性能考虑**：
- 个人数据量小（通常 < 1万条记忆）
- 评分管道延迟 < 200ms 完全可接受
- 无需复杂的性能优化

### 5.3 主动整合机制

**触发方式**：
- 定时检查（每30分钟）
- 或累积阈值（10条未整合记忆）

**实现流程**：
1. 查询未整合记忆（`WHERE consolidated = 0`）
2. 调用 LLM 分析关联
3. 更新 connections 表
4. 生成洞见记忆
5. 标记源记忆为已整合

**数据库事务**：
- SQLite 支持完整的 ACID 事务
- 整合过程在事务中执行，保证一致性

## 六、性能与扩展性

### 6.1 性能预估

**个人使用场景**：
- 记忆数量：1000 - 10000 条
- 向量维度：1024
- 数据库大小：100MB - 1GB

**性能指标**：
- 向量检索：< 50ms（暴力搜索）
- BM25 检索：< 20ms（FTS5）
- 完整检索（含12阶段）：< 200ms
- 存储记忆：< 100ms

**对比 PostgreSQL**：
- PostgreSQL：更快（HNSW 索引），但需要服务端
- SQLite：稍慢，但对个人使用完全够用

### 6.2 扩展性考虑

**数据量增长**：
- 1万条以内：暴力搜索
- 1万-10万条：使用 IVF 索引
- 10万条以上：考虑迁移到 PostgreSQL（但个人使用很少达到）

**多设备同步**：
- SQLite 单文件，方便备份和同步
- 可通过云盘（Dropbox/iCloud）同步
- 或实现简单的同步协议

## 七、部署形态对比

### 7.1 三种部署模式

| 模式 | 适用场景 | 优势 | 劣势 |
|------|---------|------|------|
| **单机模式** | 个人电脑 | 简单、快速、隐私 | 无法跨设备访问 |
| **局域网模式** | 家庭/办公室 | 多设备共享 | 需要配置网络 |
| **云端模式** | 随时随地访问 | 跨设备、高可用 | 隐私风险、成本 |

**推荐**：
- 主力：单机模式（个人电脑）
- 辅助：局域网模式（手机通过 WiFi 访问电脑）
- 可选：云端模式（部署到个人 VPS）

### 7.2 与 Docker 方案对比

| 维度 | Docker 方案 | 单一二进制方案 |
|------|------------|---------------|
| 安装复杂度 | 高（需装 Docker） | 低（下载即用） |
| 启动速度 | 慢（容器启动） | 快（秒级） |
| 资源占用 | 高（容器开销） | 低（原生进程） |
| 手机端支持 | ❌ 几乎不可能 | ✅ 完全支持 |
| 数据备份 | 需要导出卷 | 单文件拷贝 |
| 适用人群 | 技术用户 | 所有用户 |

**结论**：边端场景下，单一二进制方案明显更优。

## 八、技术选型总结

### 8.1 核心决策

| 技术点 | 原方案 | 新方案 | 理由 |
|--------|--------|--------|------|
| 数据库 | PostgreSQL | SQLite | 嵌入式、跨平台、单文件 |
| 向量检索 | pgvector | sqlite-vec | 无需服务端，性能够用 |
| 全文检索 | tsvector | FTS5 | SQLite 内置，成熟稳定 |
| 部署方式 | Docker | 单一二进制 | 开箱即用，手机支持 |
| 编程语言 | Go | Go | 编译型、跨平台、高性能 |

### 8.2 权衡取舍

**牺牲**：
- 向量检索性能（HNSW → 暴力/IVF）
- 并发写入能力（多写 → 单写）
- 企业级特性（复制、分区等）

**获得**：
- 零依赖部署
- 手机端支持
- 更低的使用门槛
- 更好的隐私保护

**结论**：对于个人知识库场景，这是正确的权衡。

## 十、风险评估与缓解

### 10.1 技术风险

**风险1：SQLite 性能不足**
- 概率：中
- 影响：高
- 缓解措施：
  - 阶段1 提前验证性能
  - 如果不满足，考虑混合方案（SQLite + 外部向量库）
  - 或提供 PostgreSQL 作为高级选项

**风险2：sqlite-vec 不成熟**
- 概率：中
- 影响：高
- 缓解措施：
  - 评估替代方案（如 Faiss、Annoy）
  - 或自己实现简单的向量检索（暴力搜索）

**风险3：手机端部署困难**
- 概率：高（iOS）
- 影响：中
- 缓解措施：
  - Android 优先（Termux 方案成熟）
  - iOS 可以先通过局域网访问电脑
  - 长期考虑原生应用

**风险4：跨平台编译问题**
- 概率：中
- 影响：中
- 缓解措施：
  - 使用 CGO_ENABLED=0 编译纯 Go 版本
  - 或使用 modernc.org/sqlite（纯 Go SQLite）

### 10.2 产品风险

**风险1：用户不会用**
- 概率：高
- 影响：高
- 缓解措施：
  - 提供详细的图文教程
  - 录制视频演示
  - 提供一键安装脚本

**风险2：浏览器插件适配困难**
- 概率：中
- 影响：中
- 缓解措施：
  - 先支持主流网站（ChatGPT、Claude）
  - 提供通用适配器框架
  - 社区贡献更多适配器

**风险3：LLM 成本过高**
- 概率：中
- 影响：中
- 缓解措施：
  - 使用便宜的模型（gpt-4o-mini）
  - 提供本地 LLM 选项（Ollama）
  - 批量处理降低调用次数

## 十一、方案总结

### 11.1 核心优势

1. **零依赖部署**：单一二进制 + SQLite，无需安装任何服务
2. **真正的跨平台**：Windows/macOS/Linux/Android 全支持
3. **隐私优先**：数据完全本地，不上传云端
4. **开箱即用**：下载即用，普通用户也能轻松使用
5. **轻量高效**：资源占用低，启动速度快

### 11.2 与原方案对比

| 维度 | PostgreSQL + Docker 方案 | SQLite + 单一二进制方案 |
|------|-------------------------|------------------------|
| 部署复杂度 | 高 | 低 |
| 手机端支持 | ❌ | ✅ |
| 性能 | 更高 | 够用 |
| 适用人群 | 技术用户 | 所有用户 |
| 维护成本 | 高 | 低 |
| 扩展性 | 强 | 中等 |

**结论**：边端场景下，SQLite 方案是更优选择。
